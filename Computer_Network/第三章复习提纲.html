〇、传输层概述（铺垫）

通俗类比（课件原例）：Ann 家 12 个孩子给 Bill 家 12 个孩子写信。主机 = 房屋，进程 = 孩子，应用消息 = 信件，传输层协议 = Ann 和 Bill（负责把信分发给家里的兄弟姐妹），网络层协议 = 邮政服务。
  • 网络层：提供主机之间的逻辑通信；
  • 传输层：提供进程之间的逻辑通信（依靠并增强网络层服务）★必背辨析；
  • 传输协议运行在端系统中：发送方把应用消息分成报文段（segment）交给网络层；接收方把报文段重组成消息交给应用层；
  • 两大传输协议：
    • TCP：可靠、有序传输；拥塞控制、流量控制、连接建立；
    • UDP：不可靠、无序交付；是"尽力而为" IP 的简单扩展；
    • 二者都不提供：延迟保证、带宽保证。

一、多路复用与多路分解（multiplexing / demultiplexing）

通俗理解：一台主机同时跑着微信、浏览器、游戏，网卡收到一堆数据，怎么知道哪个包给哪个应用？靠传输层"分拣"——这就是复用/分解。

1.1 定义 ★必背

  • 多路复用（发送方）：处理来自多个套接字的数据，添加传输层头部（头部信息稍后用于分解）；
  • 多路分解（接收方）：使用传输层头部信息将收到的报文段送到正确的套接字。

1.2 分解怎么工作

  • 主机收到 IP 数据报：每个数据报有源 IP、目的 IP，携带一个传输层报文段；报文段中有源端口号、目的端口号；
  • 主机用 IP 地址 + 端口号把报文段定向到正确的套接字。

1.3 无连接分解（UDP）vs 面向连接分解（TCP）★必考辨析

UDP（无连接）TCP（面向连接）
套接字标识二元组：(目的 IP, 目的端口)四元组：(源 IP, 源端口, 目的 IP, 目的端口)
分解依据仅目的端口号全部四个值
结果目的端口相同但源不同的数据报 → 进同一个套接字四元组不同 → 进不同套接字；服务器可同时支持多个 TCP 套接字（每个对应一个客户端）

例：三个报文段目的 IP 都是 B、目的端口都是 80，但源 IP/端口不同 → TCP 下被分解到三个不同的套接字。

二、无连接传输：UDP

2.1 UDP 特点 ★必背

  • "简化版/基础版"传输协议，尽力而为（best effort）服务，报文段可能丢失、乱序交付；
  • 无连接：无握手（省 1 个 RTT），每个 UDP 段独立处理；
  • 发送者和接收者之间没有连接状态；
  • 头部小（仅 8 字节）；
  • 没有拥塞控制：可以尽可能快地发送；拥塞时仍能工作。

2.2 为什么要有 UDP？/ 使用场景 ★常考简答

  • 无需建立连接（减少 RTT 延迟）、简单、头部开销小、不限速；
  • 场景：流媒体（容忍丢失、对速率敏感）、DNS、SNMP、HTTP/3；
  • 若需要在 UDP 上实现可靠传输：在应用层添加可靠性和拥塞控制（HTTP/3 就是这样做的）。

2.3 UDP 段格式 ★

头部只有 4 个字段、共 8 字节：源端口号、目的端口号、长度（含头部的段长，字节）、校验和 + 应用数据（有效载荷）。

2.4 UDP 校验和 ★含计算

  • 目标：检测传输中的位错误（位翻转）；
  • 发送方：把 UDP 段内容（含头部字段和 IP 地址）视作一串 16 位整数，求反码和（加法时最高位的进位要"回绕"加到结果上），结果取反放入校验和字段；
  • 接收方：重新计算并与校验和字段比较——不等 → 检测到错误；相等 → 未检测到错误，但仍可能有错（弱保护：多个位同时翻转可能校验和不变）★。

三、可靠数据传输原理（rdt）★不考状态机，考机制思想

通俗理解：网络这条"信道"不可靠——会出错、会丢包。可靠传输协议（rdt）就是在不可靠信道上"打补丁"，逐步加机制：校验和 → 确认 → 序号 → 定时器 → 流水线。考试不要求画/读状态机，但要懂每一版解决什么问题、用什么机制。

3.1 渐进式设计思路 ★核心必背

版本信道假设新增机制
rdt1.0完全可靠（无错、无丢包）什么都不用做，直接发
rdt2.0可能位错误校验和检错 + ACK（收对了）/NAK（收错了），收到 NAK 重传——"停止-等待"
rdt2.1ACK/NAK 本身也可能损坏序列号（0 和 1 即可）：ACK/NAK 损坏就重传当前分组，接收方靠序号丢弃重复分组
rdt2.2同 2.1无 NAK：接收方只发 ACK，但 ACK 必须带上被确认分组的序号；发送方收到重复 ACK 就重传（TCP 采用此思路）
rdt3.0还会丢包（数据或 ACK 都可能丢）倒计时定时器（超时重传）：等"合理"时间没收到 ACK 就重发；若只是延迟而非丢失，重传产生重复——序号已能处理
记忆口诀：检错靠校验和，反馈靠 ACK，去重靠序号，丢包靠超时

3.2 rdt2.0 为何有致命缺陷？★

如果 ACK/NAK 本身损坏，发送方不知道接收方的情况；不能盲目重传——会产生重复分组。解决：加序列号，接收方丢弃重复。

3.3 停止-等待协议的性能问题 ★经典计算

  • 停止-等待：发一个分组，等到确认再发下一个；
  • 课件例子：1 Gbps 链路、单向 15 ms（RTT 30ms）、8000 bit 分组：
    • d传输 = L/R = 8000/10⁹ = 8 μs（0.008 ms）；
    • U = 0.008 / (30 + 0.008) ≈ 0.00027——协议把千兆链路用成了"龟速"，协议限制了物理资源的性能！
U发送者 = (L/R) ÷ (RTT + L/R)

3.4 流水线（pipelining）★

  • 发送方允许多个"在途"（已发送、未确认）的分组；
  • 代价：序列号范围要增大、发送方和/或接收方要缓冲；
  • 效果：3 个分组流水线 → 利用率提升 3 倍（3L/R ÷ (RTT + L/R)）。

3.5 两种流水线协议：GBN vs SR ★必考对比

回退 N 帧（Go-Back-N，GBN）

  • 发送方：窗口最多 N 个已发送未确认分组；累积确认 ACK(n)——确认直到 n（含 n）的所有分组；只为最旧的在途分组设一个定时器；超时(n)：重传 n 及窗口内所有更高序号的分组（"回退"）；
  • 接收方：只发按序到达的最高序号的 ACK（可能产生重复 ACK）；乱序分组：可丢弃（也可缓冲，实现决定），并重发最高按序 ACK；只需记住 rcv_base。

选择性重传（Selective Repeat，SR）

  • 接收方：逐个确认所有正确收到的分组；缓冲乱序分组，等补齐后按序交付上层；
  • 发送方：为每个未确认分组单独维护定时器；超时只重传那一个分组；
  • 窗口：N 个连续序号，限制在途分组。
GBNSR
确认方式累积确认逐个（独立）确认
定时器1 个（最旧未确认分组）每个未确认分组 1 个
超时重传重传 n 及之后所有只重传超时的那一个
接收方乱序处理通常丢弃 + 重发旧 ACK缓冲

3.6 SR 的窗口大小困境 ★常考结论

序号空间有限时（如 0,1,2,3），若窗口太大，接收方无法区分"新分组"和"旧分组的重传"（两种场景下接收方行为完全相同但结果一个对一个错）。

发送窗口 + 接收窗口 ≤ 2ⁿ（n 为序号位数），且通常发送窗口 = 接收窗口

课件扩展例（协议 6）：n=3（序号 0~7），若 WT=WR=7，ACK 丢失后窗口调整前后有效序号重叠，会把重传的旧帧当新帧收下 → 协议失败；所以 WT=WR=4 才安全。

四、面向连接的传输：TCP

4.1 TCP 概述 ★必背特性清单

  • 点对点：一个发送方、一个接收方；
  • 可靠、有序的字节流：无"消息边界"；
  • 全双工：同一连接双向数据流；MSS：最大段大小；
  • 面向连接：数据交换前先握手（初始化双方状态）；
  • 流水线 + 累积确认：窗口大小由拥塞控制和流量控制共同决定；
  • 流量控制：发送方不压垮接收方。

4.2 TCP 段结构 ★

32 位一行：源端口 + 目的端口 → 序列号 → 确认号 → 头部长度 + 标志位（C、E：拥塞通知；RST、SYN、FIN：连接管理；ACK 位）+ 接收窗口（rwnd，流量控制：接收方愿意接收的字节数） → 校验和 + 紧急指针 → 选项 → 应用数据。

4.3 序列号与确认号 ★必考

  • 序列号：段中第一个数据字节在字节流中的编号（按字节计数，不是按段）；
  • 确认号：期望从对方收到的下一个字节的序号；累积确认；
  • 乱序段怎么处理？TCP 规范未规定，由实现者决定；
  • 经典 telnet 例：A 发 Seq=42, ACK=79, data='C' → B 回 Seq=79, ACK=43, data='C'（确认收到 C 并回显）→ A 回 Seq=43, ACK=80。

4.4 RTT 估计与超时设置 ★公式必背

超时值应大于 RTT，但 RTT 是变化的：太短 → 过早超时、不必要重传；太长 → 对丢失反应迟钝。

  • SampleRTT：从发送段到收到 ACK 的实测时间（忽略重传）；
  • 平滑估计（EWMA 指数加权移动平均，典型 α = 0.125）；
  • 偏差估计 DevRTT（典型 β = 0.25）。
EstimatedRTT = (1−α)·EstimatedRTT + α·SampleRTT
DevRTT = (1−β)·DevRTT + β·|SampleRTT − EstimatedRTT|
TimeoutInterval = EstimatedRTT + 4·DevRTT

4.5 TCP 可靠传输机制（发送方/接收方行为）★

发送方三事件：

  1. 收到应用数据：创建段（序号 = 第一个字节的流编号），定时器没开就开（针对最旧未确认段）；
  2. 超时：重传导致超时的段，重启定时器；
  3. 收到 ACK：若确认了新数据，更新 SendBase；仍有未确认段则重启定时器。

接收方 ACK 生成规则（RFC 5681）★常考表

接收端事件接收方动作
按序段到达，且之前都已确认延迟 ACK：最多等 500ms 看有无下一段，没有就发 ACK
按序段到达，且有另一段等待确认立即发送单个累积 ACK（一次确认两个段）
乱序段到达（序号高于期望，检测到间隙）立即发重复 ACK，指明期望的下一个字节
到达的段部分/完全填补间隙若段从间隙低端开始，立即发 ACK

4.6 快速重传 ★必背

收到 3 个重复 ACK（即同一数据的第 4 个 ACK）→ 说明该段很可能丢了 → 不等超时，立即重传最小序号的未确认段。

4.7 TCP 流量控制 ★必背、勿与拥塞控制混淆

  • 问题：网络层交付数据比应用层从套接字缓冲区取数据快 → 接收缓冲区会溢出；
  • 机制：接收方在 TCP 头部的 rwnd 字段中"通告"空闲缓冲区空间（RcvBuffer 默认约 4096 字节，许多 OS 自动调整）；
  • 发送方限制在途（未确认）数据量 ≤ rwnd，保证接收缓冲区不溢出；
  • 一句话：接收方控制发送方，别发太多太快。

4.8 TCP 连接管理 ★必考

为什么两次握手不行？★

  • 网络中有可变延迟、消息重传（如 req_conn 重传）、乱序，双方"看不见"对方；
  • 两次握手的失败场景：① 半开连接（客户端已结束，服务器还傻等）；② 重复数据被当成新数据接收。

三次握手（必背流程）

  1. 客户端发 SYN（SYNbit=1，Seq=x），状态 SYNSENT；
  2. 服务器回 SYNACK（SYNbit=1，Seq=y，ACKnum=x+1），状态 SYN RCVD——"服务器活着"；
  3. 客户端回 ACK（ACKnum=y+1），可携带数据，双方进入 ESTAB——"客户端活着"。
人类版三次握手："完成了吗？""已完成。""开始爬。"

关闭连接（四次挥手）

  • 双方各自关闭自己一端：发送 FIN=1 的段，对收到的 FIN 回 ACK（ACK 可与自己的 FIN 合并）；可处理同时 FIN；
  • 客户端最后进入 TIMED_WAIT（等 2 倍最大段生存期）再 CLOSED。

五、拥塞控制原理

5.1 什么是拥塞 ★必背

  • 非正式定义：太多源以过快速度发送过多数据，超出网络处理能力；
  • 表现：长延迟（路由器缓冲区排队）、丢包（缓冲区溢出）。
★必考区别：拥塞控制＝太多发送者发太快 → 保护网络；流量控制＝一个发送者对一个接收者太快 → 保护接收方缓冲区。

5.2 拥塞的代价（三个场景的结论）★

  • 场景 1（无限缓冲）：到达率 λin 接近链路容量 R/2 时，时延急剧增大；
  • 场景 2（有限缓冲 + 重传）：代价 1——丢包导致重传，"过早超时"还会发不必要的重复副本，重传浪费容量，有效吞吐量下降；
  • 场景 3（多跳路径）：代价 2——分组被下游丢弃时，它在上游消耗的传输容量和缓冲全部白费；极端时某流吞吐量趋于 0。

反馈来源无网络显式反馈，通过观察丢包、延迟推断拥塞路由器直接提供反馈（可指示拥塞级别或设置速率）

六、TCP 拥塞控制

6.1 AIMD（加性增、乘性减）★核心必背

  • 思路：发送方逐步加速直到丢包（说明拥塞），丢包时降速；
  • 加性增（Additive Increase）：每个 RTT 把发送速率（cwnd）增加 1 个 MSS，直到检测到丢包——"探测带宽"；
  • 乘性减（Multiplicative Decrease）：丢包时把速率减半；
  • 图像是著名的锯齿状曲线；
  • 丢包的两种处理（必背区别）：
    • 3 个重复 ACK（TCP Reno）：cwnd 减半（进快速恢复）；
    • 超时（TCP Tahoe 对所有丢包都这样）：cwnd 降到 1 MSS；
  • 为什么用 AIMD？分布式异步算法，被证明能优化全网流速率且有理想稳定性。

6.2 拥塞窗口 cwnd 与发送速率 ★公式

  • 发送方限制：LastByteSent − LastByteAcked ≤ cwnd；
TCP 速率 ≈ cwnd / RTT（字节/秒）

6.3 慢启动（slow start）★必背

  • 连接刚开始：cwnd = 1 MSS，每个 RTT 翻倍（每收到一个 ACK 就 cwnd += MSS）→ 指数增长；
  • 名字叫"慢"启动，实际初始慢但指数级增长（并不慢）。

6.4 从慢启动到拥塞避免 ★必背

  • 问：指数增长何时转为线性？答：当 cwnd 达到 ssthresh（上次丢包前 cwnd 值的一半）时；
  • 变量 ssthresh：每次丢包事件发生时，ssthresh = 丢包前 cwnd 的一半。

6.5 拥塞控制全流程总结（不要求 FSM，记行为）★

事件动作
初始cwnd = 1 MSS，ssthresh = 64 KB，慢启动
慢启动中收到新 ACKcwnd += MSS（每 RTT 翻倍）
cwnd ≥ ssthresh进入拥塞避免：每 RTT cwnd 增加 1 个 MSS
超时ssthresh = cwnd/2；cwnd = 1 MSS；重传缺失段；回到慢启动
3 个重复 ACK（Reno）ssthresh = cwnd/2；cwnd = ssthresh + 3；重传缺失段；进入快速恢复
快速恢复中收到新 ACKcwnd = ssthresh，回到拥塞避免

6.6 TCP CUBIC（了解）

  • 直觉：丢包后瓶颈链路的拥塞状态可能没大变 → 减半后先快速爬回 Wmax 附近，再缓慢谨慎逼近（W 是与时间 K 距离的立方函数）；
  • Linux 默认的 TCP 拥塞控制，流行 Web 服务器最常用。

6.7 基于延迟的拥塞控制 / BBR（了解）

  • 目标："保持管道刚好满，但不过满"——瓶颈链路持续传输又不造成大排队；
  • 方法：用 RTTmin（无拥塞 RTT）算出无拥塞吞吐量 = cwnd/RTTmin；实测吞吐量接近它 → 线性增 cwnd；远低于它 → 线性减；
  • 不靠丢包驱动；Google 内部骨干网部署 BBR。

6.8 TCP 吞吐量公式（了解，可能考代入）

平均 TCP 吞吐量 = (3/4)·W/RTT（字节/秒）
  • 忽略慢启动，W 为丢包时窗口（窗口在 W/2 和 W 之间锯齿，平均 3W/4）；
  • 长肥管道公式 [Mathis 1997]：吞吐量 = 1.22·MSS / (RTT·√L)（L 为丢包率）——10 Gbps 需要 L ≈ 2×10⁻¹⁰，极小！

6.9 ECN 显式拥塞通知（了解）

网络辅助式：路由器在 IP 头部 ToS 字段的 2 个比特标记拥塞（ECN=11）→ 接收方在 TCP ACK 段设 ECE 位通知发送方 → 同时涉及 IP 层和 TCP 层。

6.10 TCP 公平性 ★常考结论

  • 目标：K 条 TCP 会话共享带宽 R 的瓶颈链路，每条平均得 R/K；
  • TCP 公平吗？是（理想假设下：相同 RTT、固定数量会话且仅处于拥塞避免阶段）——AIMD 几何解释：加性增沿 45° 线，乘性减向原点收缩，最终收敛到平等共享线；
  • 不公平的现实：
    • UDP：多媒体应用不用 TCP、不限速，没有"互联网警察"；
    • 并行 TCP 连接：应用开多条连接抢带宽（链路已有 9 条连接时：新应用开 1 条得 R/10，开 11 条得 R/2）。

七、传输层功能的演进 / QUIC（了解）

  • TCP/UDP 用了 40 年；特殊场景有特殊"风味"TCP：长肥管道、无线网络（误码丢包被 TCP 误判为拥塞）、长延迟链路、数据中心、后台流量；
  • QUIC（Quick UDP Internet Connections）：基于 UDP 的应用层协议，HTTP/3 的底层：
    • 1 次握手同时建立可靠性、拥塞控制、认证、加密状态（TCP+TLS 要 2 次串行握手）；
    • 单连接上多路复用多个应用级"流"：每流独立的可靠传输→ 消除队头阻塞（HOL）；
    • 错误/拥塞控制算法与 TCP 类似。

8.4 易混概念辨析

概念对区别一句话
网络层 vs 传输层网络层：主机之间逻辑通信；传输层：进程之间逻辑通信
UDP 分解 vs TCP 分解UDP 只看（目的 IP, 目的端口）二元组；TCP 看完整四元组
流量控制 vs 拥塞控制流控：别压垮接收方（rwnd，接收方通告）；拥控：别压垮网络（cwnd，发送方自己推断）
GBN vs SRGBN 累积确认、1 个定时器、超时全部重发、乱序丢弃；SR 独立确认、每段定时器、只重发丢的、乱序缓冲
超时重传 vs 快速重传超时：定时器到期才重传；快速重传：3 个重复 ACK 提前重传，不等超时
Tahoe vs RenoTahoe：任何丢包 cwnd 都归 1；Reno：3 重复 ACK 只减半（快速恢复），超时才归 1
慢启动 vs 拥塞避免慢启动：指数增长（每 RTT 翻倍）；拥塞避免：线性增长（每 RTT 加 1 MSS）；分界点 ssthresh
序列号 vs 确认号序号：本段第一个字节的编号；确认号：期望对方发来的下一个字节编号（累积）
两次握手 vs 三次握手两次无法应对延迟/重传/乱序 → 半开连接、重复数据；第三次 ACK 证明"客户端活着"
