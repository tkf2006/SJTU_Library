〇、控制平面概述（铺垫）

  • 回顾：转发 = 数据平面（把分组从输入端口移到输出端口）；路由 = 控制平面（确定分组从源到目的的路径）；
  • 构建控制平面的两种方法 ★必背：
    1. 每路由器控制（per-router control，传统）：每台路由器里的路由算法组件相互交互，各自算出转发表；
    2. 逻辑集中控制（SDN）：远程控制器计算转发表并安装到各路由器。

一、路由协议（路由算法）

1.1 基本概念与图抽象 ★

  • 路由协议目标：确定从发送主机到接收主机经过路由器网络的"优质"路径（成本最低/最快/最不拥塞）；
  • 路径 = 分组从源到目的所经过的路由器序列；
  • 图抽象：G = (N, E)，N = 路由器集合，E = 链路集合；ca,b = a、b 间链路成本（非直连 = ∞）；成本由运营商定义（可以全为 1，或与带宽成反比、与拥塞相关）。

1.2 路由算法分类 ★必背

分类维度类型
信息是全局还是分散？全局（所有路由器掌握完整拓扑和链路成本）→ 链路状态 LS 算法；去中心化（只知道邻居，迭代交换信息）→ 距离向量 DV 算法
路由变化快慢？静态（变化慢）vs 动态（定期更新/响应链路成本变化）

课件补充分类：静态算法（Dijkstra、泛洪）、自适应算法（LS、DV）、拓扑相关路由、移动节点路由、Ad-hoc 路由。

1.3 链路状态算法（LS）与 Dijkstra ★必考会算

LS 基本思想（五步）：① 发现邻居并获取地址；② 测量到各邻居的成本；③ 组装链路状态分组；④ 发送（泛洪）给所有其他路由器；⑤ 用 Dijkstra 计算到所有节点的最短路径。

Dijkstra 算法要点：

  • 集中式：全网拓扑、链路成本对所有节点已知（靠"链路状态广播"，所有节点信息相同）；
  • 计算一个节点（源）到所有其他节点的最低成本路径 → 得到该节点的转发表；
  • 迭代式：k 次迭代后，知道到 k 个目的地的最低成本路径；
  • 注意：边数最少的路径不一定是最短路径！

记号：D(v) = 当前到 v 的最小成本估计；p(v) = 前驱节点；N' = 已确定最小成本路径的节点集合。

算法流程（必会手算）：

  1. 初始化：N'={u}；u 的邻居 D(v)=cu,v，其余 ∞；
  2. 循环：找不在 N' 中 D(w) 最小的 w，加入 N'；
  3. 对 w 的所有不在 N' 中的邻居 v 更新；
  4. 直到所有节点都在 N' 中；由前驱指针回溯得最小成本路径树和转发表。
D(v) = min( D(v), D(w) + cw,v )

复杂度 ★：n 个节点，n(n+1)/2 次比较 → O(n²)（堆实现可 O(n log n)）；消息复杂度：每个路由器广播链路状态给其他 n 个路由器 → O(n²) 条消息。

振荡问题：当链路成本依赖于流量时，LS 可能出现路由振荡；防止方法：成本不依赖流量、异步算法、随机通告。

1.4 距离向量算法（DV）★必考会算

基于 Bellman-Ford 方程（动态规划）：

Dx(y) = minv { cx,v + Dv(y) }　（对 x 的所有邻居 v 取最小）

含义：x 到 y 的最小成本 = "到某邻居 v 的直接成本 + v 到 y 的最小成本"中的最小值；取得最小值的邻居 v 就是下一跳。

关键思想：

  • 每个节点不时把自己的距离向量（DV）估计发给邻居；
  • 收到邻居的 DV 后，用 B-F 方程更新自己的 DV；
  • 在自然条件下，估计值收敛到实际最小成本。

三个特征词 ★：迭代、异步（由本地链路变化或邻居 DV 消息触发）、分布式、自停止（只有自己的 DV 变化才通知邻居——"没消息就是好消息"）。

经典计算例：Du(z) = min{ cu,v+Dv(z), cu,x+Dx(z), cu,w+Dw(z) } = min{2+5, 1+3, 5+3} = 4，经 x 走。

链路成本变化：

  • "好消息传得快"：成本变小，节点更新后立即通知邻居，逐跳扩散；
  • "坏消息传得慢"——计数到无穷问题 ★必考：链路成本变大（如 4→60）时，y 误用 z 经过自己的旧路径信息，于是 y、z 互相"接力"+1，慢慢数到无穷；
  • 缓解方法 ★：
    • 水平分裂（split horizon）/毒性逆转：对下游节点通告真实距离，对上游（路径经过对方的）节点通告无穷大；
    • Holddown：发现路径不通时不立即换路，先置为无穷大。

DV 缺点：交换信息量大、信息可能不一致、收敛慢（坏消息）、不适合大型网络。

1.5 LS vs DV 比较 ★必考

LS（链路状态）DV（距离向量）
信息全局拓扑只与邻居交换
消息复杂度O(n²) 条消息邻居间交换，收敛时间不定
收敛O(n²) 算法；可能振荡收敛慢；可能路由环路、计数到无穷
鲁棒性路由器可能通告错误链路成本，但每台只算自己的表，错误影响小可能通告错误路径成本（"我到哪都超便宜"→ 黑洞）；表被他人使用，错误会扩散全网

1.6 其他路由方式（课件补充，了解）

  • 泛洪/扩散法：不算路径，见路就转；用跳数计数器或登记表抑制重复；优点可靠、路径最短（军用），缺点重复分组多、浪费带宽；
  • 分层路由：路由器分区域，区域外抽象成一个节点，大幅缩短路由表（N 个路由器最优层数 lnN，每台 e·lnN 个表项）；
  • 广播路由：多目的地路由、生成树法、逆向路径转发（RPF）——分组从"通向广播源的正常线路"来才转发，否则丢弃；
  • 多播路由：用 D 类地址；生成树法（每组每成员一棵，存储大）vs 核心树（每组一棵，树根在中心）；
  • P2P 节点查找：Chord：节点和资源名都散列到 0~2m−1 的环上，资源索引存在 successor(key)，每节点维护 finger 表加速查找；
  • 移动 IP：移动节点、主代理（home agent）、客代理（foreign agent）；数据包先到主网络，主代理隧道封装转给客代理，再交移动主机；
  • Ad-hoc 路由：主动/表驱动（DSDV）vs 按需/事件驱动（AODV：RREQ 广播 + RREP 沿逆向路径单播，假设链路双向；DSR：源路由 + Route Cache，支持单向链路、可存多条路径）。

二、自治系统内部路由：OSPF

2.1 为什么要分自治系统（AS）？★必背

理想化假设（所有路由器相同、网络"扁平"）不现实：

  1. 规模：数十亿目的地，路由表存不下，交换路由表会淹没链路；
  2. 管理自治：互联网 = 网络的网络，每个网络管理员想自己控制内部路由。

→ 把路由器聚合成自治系统（AS，autonomous systems，也称域 domain）：

  • 域内（intra-AS）路由：同一 AS 内；AS 内所有路由器必须跑同一个域内协议；不同 AS 可以跑不同协议；
  • 域间（inter-AS）路由：AS 之间；
  • 网关路由器（gateway router）：位于 AS"边缘"，连接其他 AS，同时执行域内+域间路由；
  • 转发表由域内 + 域间路由算法共同配置 ★：AS 内目的地的表项由域内路由决定；外部目的地的表项由域间与域内共同决定。

2.2 常见域内路由协议

  • RIP：经典距离向量，每 30 秒交换一次 DV（已不常用）；
  • EIGRP：基于 DV，思科专有（后开放）；
  • OSPF（Open Shortest Path First，开放最短路径优先）：链路状态，最常用；IS-IS 与之本质相同。

2.3 OSPF 要点 ★必背

  • "开放"：公开可用；
  • 经典链路状态：每台路由器把链路状态通告泛洪到全 AS（直接承载在 IP 上，不用 TCP/UDP）；每台路由器有完整拓扑，用 Dijkstra 算转发表；
  • 链路成本度量多样：带宽、延迟等；
  • 安全：所有 OSPF 消息经过认证（防恶意入侵）。

2.4 分层 OSPF ★

  • 两级层次：多个区域（area）+ 骨干（backbone）；
  • 链路状态通告只在区域内（或骨干内）泛洪；每个节点只有本区域详细拓扑，对其他区域只知道"方向"；
  • 角色：内部路由器（区域内泛洪和计算）、区域边界路由器（汇总到本区域的距离、在骨干上通告）、骨干路由器（只跑骨干 OSPF）、边界路由器（连其他 AS）。

三、自治系统间路由：BGP

3.1 BGP 是什么 ★必背

  • BGP（Border Gateway Protocol，边界网关协议）：事实上的域间路由协议——"把互联网粘在一起的胶水"；
  • 允许子网向互联网其余部分宣告自己的存在："我在这里，这是我能到达的地方"；
  • 提供两种手段：
    • eBGP：从相邻 AS 获取子网可达性信息；
    • iBGP：把可达性信息传播到 AS 内部所有路由器；
  • 再根据可达性信息和策略选择"好"路径；网关路由器同时跑 eBGP 和 iBGP。

3.2 BGP 基础 ★

  • BGP 会话：两个 BGP 路由器（"对等体 peer"）通过半永久 TCP 连接交换 BGP 消息；
  • BGP 是"路径向量"协议：通告到目的前缀的完整路径；
  • 当 AS3 网关向 AS2 网关通告路径 "AS3, X" 时，意味着 AS3 承诺会把数据报转发到 X。

3.3 路径属性与策略路由 ★必背

BGP 通告的路由 = 前缀（prefix）+ 属性（attributes），两个重要属性：

属性含义
AS-PATH通告所经过的 AS 列表
NEXT-HOP通向下一跳 AS 的特定内部路由器

基于策略的路由：网关收到通告后用导入策略决定接受/拒绝（如"绝不经过 AS Y"）；AS 还决定是否把路径再通告给其他邻居 AS。

通告传播例：3a --eBGP--> 2c 通告 "AS3,X" → 2c 经 iBGP 传遍 AS2 → 2a 经 eBGP 向 1c 通告 "AS2,AS3,X"；1c 可能同时从 3a 学到 "AS3,X"，按策略选择其一并经 iBGP 传给 AS1 内部。

3.4 BGP 消息四种 ★

消息作用
OPEN与远程对等体建立 TCP 连接并认证
UPDATE通告新路径（或撤销旧路径）
KEEPALIVE无 UPDATE 时保活；也用于确认 OPEN
NOTIFICATION报告之前消息中的错误；也用于关闭连接

3.5 BGP + 域内路由 → 转发表（必懂流程）

1a、1b、1d 经 iBGP 学到"到 X 走 1c" → 各自用 OSPF 域内路由算出"到 1c 走哪个接口" → 在转发表里写"目的 X → 该接口"。

3.6 热土豆路由（hot potato routing）★必背

当可以从多个网关离开本 AS 时，选择域内成本最小的本地网关（"赶紧把烫手山芋扔出去"）——不考虑域间成本（即使 AS 跳数更多）。

3.7 策略实例 ★常考

背景：ISP 只想为自己的客户流量服务，不想给其他 ISP 当"免费中转"。

  • 例 1：A 向 B、C 通告路径 Aw；B 选择不向 C 通告 BAw——因为 C、A、w 都不是 B 的客户，B 转发这些流量没有收益 → C 学不到 CBAw，只能走 CAw；
  • 例 2：x 双宿（连 B 和 C）：x 不想替 B 到 C 中转流量 → x 不向 B 通告它到 C 的路由。

3.8 BGP 路由选择顺序 ★必背

  1. 本地偏好值（local preference，策略决定）；
  2. 最短 AS-PATH；
  3. 最近的 NEXT-HOP 路由器（热土豆路由）；
  4. 附加标准。

3.9 为什么域内、域间用不同协议？★常考简答

维度域内域间
策略单一管理员，策略问题少管理员要控制流量如何路由、谁能借道 → 策略优先
规模分层路由缩表、减少更新流量同样需要应对规模
性能可以专注性能策略优先于性能

四、SDN 控制平面

4.1 为什么要逻辑集中的控制平面？★必背

  • 更简单的网络管理：避免路由器配置错误，流量调度更灵活；
  • 基于表（OpenFlow API）的转发让"编程"路由器成为可能：集中计算+分发表，比分布式协议简单；
  • 开放（非专有）实现：促进创新。

传统路由难以实现的流量工程 ★（三个经典问题）：

  1. 想让 u→z 流量走 uvwz 而非 uxyz？只能改链路权重——"旋钮"控制力有限；
  2. 想把 u→z 流量在两条路径上分流（负载均衡）？传统路由做不到；
  3. 想让 w 对红蓝两类流量走不同路由？基于目的地转发 + LS/DV 做不到 → 需要通用转发 + SDN。

4.2 SDN 架构四大支柱 ★必背

  1. 广义的"基于流"的转发（如 OpenFlow）；
  2. 控制平面与数据平面分离；
  3. 控制平面功能放在数据平面交换机之外（远程控制器）；
  4. 可编程的控制应用。

4.3 SDN 三层结构 ★必考

层内容
网络控制应用（"控制的大脑"）路由、访问控制、负载均衡等；用控制器的 API 实现控制功能；解耦——可由第三方提供
SDN 控制器（网络操作系统）维护网络状态信息；通过北向 API 与上层应用交互；通过南向 API 与下层交换机交互；为性能/扩展性/容错用分布式系统实现
数据平面交换机快速简单的商品化硬件转发；流表由控制器计算安装；通过 OpenFlow 等协议与控制器通信

控制器内部三层：通信层（OpenFlow/SNMP 与设备通信）→ 全网状态管理层（链路/交换机/流表的分布式数据库）→ 面向应用的抽象接口层（网络图、intent、REST API）。

4.4 OpenFlow 协议 ★

  • 运行在控制器与交换机之间，用 TCP 交换消息（可选加密）；
  • 注意区分：OpenFlow 协议（控制器↔交换机通信）≠ OpenFlow API（描述通用转发动作）；
方向关键消息
控制器 → 交换机features（查询交换机特性）、configure（查/设配置）、modify-state（增删改流表条目）、packet-out（从指定端口发出分组）
交换机 → 控制器packet-in（把分组送交控制器）、flow-removed（流表条目被删）、port status（端口状态变化）
对称其他杂项

4.5 控制/数据平面交互例（链路故障，必懂六步）★

  1. s1 检测链路故障，用 port status 消息通知控制器；
  2. 控制器更新链路状态信息；
  3. 已注册"链路变化回调"的 Dijkstra 路由应用被调用；
  4. 该应用读取网络图/链路状态，计算新路由；
  5. 与流表计算组件交互生成新流表；
  6. 控制器用 OpenFlow modify-state 把新表装入受影响交换机。

4.6 具体控制器与挑战（了解）

  • OpenDaylight（ODL）：设备商推动，有服务抽象层（SAL）；ONOS：运营商推动，强调分布式核心与 intent 框架（描述"做什么"而非"怎么做"）；
  • 挑战：控制平面的可靠性/可扩展性/安全性、跨 AS 扩展、SDN 在 5G 中至关重要。

五、网络控制报文协议（ICMP）

5.1 ICMP 要点 ★必背

  • 由主机和路由器用来交换网络层信息：
    • 错误报告：主机/网络/端口/协议不可达；
    • 回显请求/应答（echo request/reply）——ping 就用它；
  • 体系位置：网络层中"位于 IP 之上"——ICMP 消息封装在 IP 数据报中传输 ★易考；
  • 消息格式：类型 + 代码 + 引发错误的 IP 数据报的前 8 字节。

5.2 常考类型/代码表 ★

类型代码描述
00回显应答（ping 应答）
30/1目标网络/主机不可达
33目标端口不可达
80回显请求（ping 请求）
110TTL 过期
120错误的 IP 头

5.3 traceroute 与 ICMP ★必考原理

  • 源向目的发送多组 UDP 段：第 1 组 TTL=1，第 2 组 TTL=2，…；
  • 第 n 组到达第 n 个路由器时：TTL 减到 0，路由器丢弃并向源发 ICMP 类型 11 代码 0（TTL 过期）消息（含路由器名和 IP）；源记录 RTT；
  • 停止条件：UDP 段最终到达目的主机，目的主机返回 ICMP 类型 3 代码 3（端口不可达） → 源停止。

六、简单网络管理协议（SNMP）与网络管理

6.1 什么是网络管理

  • 网络管理 = 部署、集成和协调硬件/软件/人员，以监控、测试、轮询、配置、分析、评估和控制网络资源；
  • 三大组件 ★：管理服务器（managing server，通常有管理员参与）、受管设备（managed device，内含代理 agent 和数据——配置数据、操作数据、统计）、网络管理协议（服务器查询/配置设备，设备上报数据/事件）；
  • 运营商三种管理方式 ★：CLI（命令行直接配置单台设备）、SNMP/MIB（查询/设置设备数据）、NETCONF/YANG（更抽象、面向全网多设备配置管理）。

6.2 SNMP 协议 ★必背

两种信息传递模式：

  • 请求/响应模式：管理服务器主动查询/设置；
  • 陷阱（trap）模式：设备主动向管理服务器报告异常事件。
消息类型功能
GetRequest / GetNextRequest / GetBulkRequest管理者→代理："给我数据"（单个/下一个/一批）
SetRequest管理者→代理：设置 MIB 值
Response代理→管理者：返回值，响应请求
Trap代理→管理者：主动通知异常事件

6.3 MIB（管理信息库）★

  • MIB = 受管设备的操作（及部分配置）数据的集合；RFC 已定义 400 个 MIB 模块，另有大量厂商私有模块；
  • 数据定义语言：SMI（管理信息结构）；
  • 例（UDP 模块的 MIB 变量）：UDPInDatagrams（收到的报文总数）、UDPNoPorts（端口无应用而无法交付数）、UDPOutDatagrams（发送总数）等。

6.4 NETCONF/YANG（了解）

  • NETCONF：管理服务器与受管设备之间，主动管理/配置全网设备；操作（get-config、edit-config、lock/unlock、订阅通知等）；RPC 范式，消息用 XML 编码，经安全可靠传输（如 TLS）；支持跨多设备原子提交；
  • YANG：数据建模语言，定义 NETCONF 数据的结构/语法/语义，可生成 XML、表达约束保证配置正确一致。

7.4 易混概念辨析

概念对区别一句话
LS vs DVLS：全局信息+Dijkstra，自己算自己的表；DV：只和邻居换信息+B-F 方程，错误会扩散（计数到无穷、黑洞）
每路由器控制 vs SDN前者分布式算法算表；后者远程控制器集中算表再下发
域内 vs 域间路由域内（OSPF）：一个 AS 内、同一协议、重性能；域间（BGP）：AS 之间、重策略
eBGP vs iBGPeBGP：从邻居 AS 学可达性；iBGP：把信息传给 AS 内部所有路由器
OSPF vs BGPOSPF 链路状态、IP 承载、算最短路径；BGP 路径向量、TCP 承载、按策略选路
AS-PATH vs NEXT-HOPAS-PATH：经过的 AS 列表（防环、选短路径）；NEXT-HOP：进入下一 AS 的具体路由器地址
热土豆路由只看域内成本最小的出口网关，不管域间成本
北向 vs 南向 API北向：控制器 ↔ 上层控制应用；南向：控制器 ↔ 下层交换机（OpenFlow）
OpenFlow 协议 vs API协议管"控制器和交换机怎么通信"；API 管"流表能匹配什么、做什么动作"
ping vs tracerouteping：ICMP 回显请求/应答（8/0）；traceroute：UDP+TTL 递增，靠 ICMP 11 报中间路由器、3/3 报终点
SNMP 请求/响应 vs trap请求/响应：管理者主动问；trap：设备主动报异常
MIB vs YANGMIB：SNMP 的设备数据集合（SMI 定义）；YANG：NETCONF 的数据建模语言

7.5 一句话带过（非重点，了解即可）

  • 泛洪、分层路由、广播/多播路由（RPF、核心树）：课件补充内容，理解基本思想即可。
  • Chord：结构化 P2P 查找，散列环 + finger 表，O(log n) 查找。
  • 移动 IP：主代理隧道转发给客代理；移动 IPv6 无客代理、支持直接通信。
  • Ad-hoc 路由：DSDV（表驱动）vs AODV/DSR（按需）；AODV 要求双向链路、每目的一条路径；DSR 源路由、Route Cache 可存多条。
  • NETCONF/YANG：面向全网的下一代配置管理（XML RPC + 数据建模）。
  • ODL/ONOS：两大开源 SDN 控制器。
