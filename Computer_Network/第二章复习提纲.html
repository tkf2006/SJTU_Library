一、应用架构（application architecture）

通俗理解：写网络应用只需要在"网络边缘"的终端系统上写程序（网络核心设备不运行用户应用程序），这些程序之间怎么组织？有两种经典套路。

1.1 客户端-服务器架构（client-server）★必考对比

角色特点
服务器始终在线的主机；永久 IP 地址；通常位于数据中心以实现扩展
客户端与服务器通信；可能间歇性连接；可能拥有动态 IP 地址；客户端之间不互相通信

例子：HTTP、IMAP、FTP。

1.2 P2P 架构（peer-to-peer，对等式网络）

  • 无始终在线的服务器；
  • 任意终端系统（对等节点）直接通信；
  • 节点请求其他节点提供服务，同时作为回报向其他节点提供服务；
  • 自我可扩展性（self-scalability）★：新节点带来新的服务需求，同时也带来新的服务能力；
  • 缺点：节点断断续续连接、IP 地址会变 → 管理复杂；
  • 例子：P2P 文件共享（BitTorrent）。

1.3 文件分发时间对比（C/S vs P2P）★经典计算题

服务器上传容量 us，节点 i 上传/下载容量 ui / di，文件大小 F，N 个节点：

C/S：D ≥ max{ NF/us , F/dmin }　——随 N 线性增长
P2P：D ≥ max{ F/us , F/dmin , NF/(us + Σui) }　——分母随 N 增大，扩展性好

1.4 BitTorrent（P2P 文件分发实例，了解）

  • 文件分成 256KB 的块（chunk）；torrent（种子）= 交换文件块的一组对等体；tracker（跟踪器）= 跟踪参与的对等体；
  • 请求块：优先请求最稀缺的块（rarest first）；
  • 发送块：以牙还牙（tit-for-tat）——把块发给当前给自己发送速率最高的 4 个节点（每 10 秒重新评估），每 30 秒随机"乐观地"解封一个新节点。

二、进程通信

2.1 基本概念 ★必背

  • 进程（process）：在主机中运行的程序；
  • 同一台主机内的两个进程：通过进程间通信（由操作系统定义）通信；
  • 不同主机上的进程：通过交换消息（messages）通信；
  • 客户端进程：发起通信的进程；服务器进程：等待被连接的进程；
  • 注意：P2P 架构的应用同时具有客户端进程和服务器进程。

2.2 应用层协议定义了什么 ★

  1. 交换的消息类型（如请求、响应）；
  2. 消息语法（消息有哪些字段、字段如何划分）；
  3. 消息语义（字段中信息的含义）；
  4. 进程何时、如何发送和响应消息的规则。

开放协议：定义在 RFC 中、人人可用、支持互操作性（HTTP、SMTP）；专有协议：Skype、Zoom。

2.3 应用程序需要什么传输服务？★四个维度

维度说明
数据完整性文件传输、网络交易需要 100% 可靠；音频可容忍丢失
时间（时延）网络电话、互动游戏需要低延迟（约 10ms 量级）
吞吐量多媒体应用需要最低吞吐量；"弹性应用"有多少用多少
安全加密、数据完整性等

2.4 TCP vs UDP 服务 ★必考对比

TCP（传输控制协议）UDP（用户数据报协议）
可靠性可靠传输不可靠传输
连接面向连接（需先建立连接）无连接
流量控制有（发送方不压垮接收方）无
拥塞控制有（网络过载时限制发送方）无
不提供时序、最低吞吐量保证、安全可靠性、流控、拥控、时序、吞吐量保证、安全、连接建立
  • TLS（传输层安全）：在应用层实现（应用使用 TLS 库），为 TCP 连接提供加密、数据完整性、端点认证——普通 TCP/UDP 套接字没有加密，明文密码直接上网！
  • 常见应用与协议：FTP/SMTP/HTTP → TCP；互联网电话（SIP、RTP）→ TCP 或 UDP；互动游戏 → UDP 或 TCP。

三、套接字（Socket）

通俗理解：套接字就像一扇"门"——发送进程把消息推出门外，剩下的交给门另一侧的传输设施送到对方的门口。
  • 进程向它的套接字发送/接收消息；
  • 通信涉及两个套接字：每侧一个；
  • 套接字是应用进程与端到端传输协议之间的接口（门）；
  • 分界线：应用层及套接字以上由应用开发人员控制，传输层及以下由操作系统控制。

套接字编程要点（两种套接字类型）

UDP 套接字（SOCK_DGRAM）TCP 套接字（SOCK_STREAM）
连接无握手，每个分组显式附带目的 IP+端口客户端先 connect，服务器 accept 后产生新套接字与该客户端通信
服务不可靠的数据报传输，可能丢失、乱序可靠的、按序的字节流（"管道"）
服务器特征一个套接字服务所有客户端欢迎套接字 + 每客户端一个连接套接字（可同时服务多个客户端）
TCP 流程：服务器 socket → bind → listen → accept；客户端 socket → connect → send/recv → close

四、进程寻址 ★必考

问：知道主机的 IP 地址，足以标识主机上的进程吗？
答：不够！同一台主机上可以运行许多进程。
进程的标识符 = IP 地址 + 端口号（port number）
  • IP 地址：32 位，唯一标识主机；
  • 端口号：标识主机上的具体进程；
  • 常用端口号必背：HTTP 服务器：80；邮件服务器（SMTP）：25；
  • 例：向 gaia.cs.umass.edu 发送 HTTP 消息 → IP 128.119.245.12 + 端口 80。

五、万维网与 HTTP 协议

5.1 Web 基本概念

  • 网页（web page）由对象（objects）组成：对象可以是 HTML 文件、JPEG 图像、Java 小程序、音频文件等，可存储在不同服务器上；
  • 网页 = 1 个基本 HTML 文件 + 若干引用对象；
  • 每个对象由 URL 寻址：www.someschool.edu/someDept/pic.gif = 主机名 + 路径名。

5.2 HTTP 概述 ★必背

  • HTTP（HyperText Transfer Protocol，超文本传输协议）：Web 的应用层协议；
  • 客户端/服务器模型：客户端 = 浏览器（请求、接收、展示对象）；服务器 = Web 服务器（响应请求发送对象）；
  • HTTP 使用 TCP：客户端向服务器 80 端口发起 TCP 连接，交换 HTTP 消息后关闭连接；
  • HTTP 是"无状态"的（stateless）★：服务器不保留关于过去客户请求的信息（维护状态的协议很复杂，崩溃后状态视图可能不一致）。

5.3 两种 HTTP 连接 ★必考

非持久 HTTP持久 HTTP（HTTP 1.1）
每条 TCP 连接最多传 1 个对象可传多个对象
下载 N 个对象需要 N 条连接1 条连接即可
每对象耗时2 RTT + 传输时间后续对象约 1 RTT
  • RTT 定义★：一个小分组从客户端到服务器再返回所需的时间；
  • 非持久 HTTP 的问题：每个对象 2 个 RTT、每条 TCP 连接有操作系统开销 → 浏览器常开多个并行 TCP 连接；
  • 持久 HTTP：服务器发送响应后保持连接开启，后续消息走同一连接，客户端遇到引用对象立即请求，响应时间减半。
非持久 HTTP 响应时间 = 2RTT + 文件传输时间

5.4 HTTP 请求消息 ★

  • 两类消息：请求（request）和响应（response）；ASCII 人类可读；
  • 请求消息结构：请求行（方法 + URL + 版本）→ 首部行（如 Host:、User-Agent:、Connection:）→ 空行 → 实体主体；
GET /index.html HTTP/1.1\r\n
Host: www-net.cs.umass.edu\r\n
User-Agent: Mozilla/5.0 ...\r\n
Connection: keep-alive\r\n
\r\n
方法用途
GET请求对象；向服务器发数据时放在 URL 的 ? 之后
POST表单输入放在实体主体中发送给服务器
HEAD与 GET 类似但只返回响应头（不要对象）
PUT上传新文件，完全替换指定 URL 处的文件

5.5 HTTP 响应消息与状态码 ★必背

响应结构：状态行（协议 + 状态码 + 状态短语）→ 首部行（Date、Server、Last-Modified、Content-Length、Content-Type 等）→ 数据。

状态码含义
200 OK请求成功，对象在消息后部
301 Moved Permanently对象已永久移动，新位置在 Location: 首部中
400 Bad Request服务器无法理解请求
404 Not Found服务器上找不到请求的文档
505 HTTP Version Not SupportedHTTP 版本不支持

5.6 HTTP/2 与 HTTP/3（了解）

  • HTTP 1.1 的问题：流水线 GET 按 FCFS（先来先服务）响应，小对象排在大对象后面要等——队头阻塞（HOL blocking）；
  • HTTP/2（2015）：按客户指定的对象优先级传输（不一定 FCFS）；可推送未请求的对象；把对象划分成帧、交错传输以缓解 HOL 阻塞；
  • HTTP/2 仍在单条 TCP 上：丢包恢复仍会阻塞所有对象、无安全性 → HTTP/3：基于 UDP，增加安全性、对象级错误与拥塞控制。

六、网络 Cookie ★必考

通俗理解：HTTP 本身"没记性"（无状态），网站想认出"你又来了"，就发给你一张"会员卡"（cookie），你每次来都出示，网站就能对上号。

6.1 四个组件 ★必背

  1. HTTP 响应消息中的 cookie 头行（set-cookie: 1678）；
  2. 下一个 HTTP 请求消息中的 cookie 头行（cookie: 1678）；
  3. 用户主机上的 cookie 文件，由浏览器管理；
  4. 网站的后端数据库。

6.2 工作流程（Susan/Amazon 例子）

首次访问 → 服务器创建唯一 ID（cookie），在后端数据库建立条目，响应带 set-cookie: 1678 → 浏览器存入 cookie 文件 → 后续请求都带 cookie: 1678 → 网站"识别"出 Susan（一周后再访问仍然认识）。

6.3 用途与隐私

  • 用途：授权、购物车、推荐、用户会话状态（Web 邮件）；
  • 保持状态的两种思路：在协议端点维护状态 vs 在消息中携带状态（cookies）；
  • 第一方 cookie：来自你正在访问的网站；
  • 第三方 cookie（跟踪 cookie）★：来自你没有访问的网站（如广告商 AdX），可在多个网站间跟踪同一身份，用户可能毫无察觉（甚至用不可见链接触发）；Firefox/Safari 默认禁用，Chrome 2023 年停用；
  • GDPR（欧盟通用数据保护条例，2018.5）：cookie 能识别个人时即属个人数据，用户必须能明确控制是否允许 cookie。

七、网页缓存（Web caching）

7.1 Web 缓存 / 代理服务器 ★必背

  • 目标：不经过源服务器就满足客户端请求；
  • 用户把浏览器指向本地 Web 缓存，所有 HTTP 请求先发给缓存：命中 → 缓存直接返回对象；未命中 → 缓存向源服务器请求，收到后缓存一份再返回给客户端；
  • Web 缓存同时扮演客户端和服务器（对源服务器是客户端，对浏览器是服务器）；
  • 为什么要缓存：① 减少响应时间（缓存离客户端近）；② 减少机构接入链路的流量；③ 帮助"缺乏资源"的内容提供者有效分发内容；
  • 服务器通过响应头告诉缓存对象是否允许缓存。

7.2 缓存例题 ★经典计算

场景：接入链路 1.54 Mbps，路由器到服务器 RTT = 2 s，对象 100 Kbits，请求率 15 个/秒（即 1.50 Mbps）。

  • 无缓存：接入链路利用率 = 1.50/1.54 = 0.97 → 排队时延巨大（分钟级）；
  • 方案 1（买 154 Mbps 快链路）：利用率 0.0097，时延约 2s，但贵；
  • 方案 2（装 Web 缓存，假设命中率 0.4）：
    • 60% 请求走接入链路：流量 = 0.6 × 1.50 = 0.9 Mbps，利用率 = 0.9/1.54 = 0.58 → 排队时延小（毫秒级）；
    • 平均端到端时延 = 0.6 × 2.01s + 0.4 × 毫秒级 ≈ 1.2 秒——比买快链路更低且更便宜！

7.3 条件 GET（浏览器缓存）★

  • 目标：缓存副本是最新的就不重复传输对象；
  • 客户端请求带：If-modified-since: <date>；
  • 服务器响应：未修改 → 304 Not Modified（不含对象）；已修改 → 200 OK + 新对象。

八、电子邮件协议

8.1 三大组成部分 ★必背

  1. 用户代理（user agent）："邮件阅读器"，撰写/编辑/阅读邮件（Outlook、iPhone 邮件客户端）；
  2. 邮件服务器（mail server）：邮箱（mailbox）存传入消息，消息队列存待发送消息；
  3. SMTP（Simple Mail Transfer Protocol，简单邮件传输协议）：邮件服务器之间传送邮件的协议。

8.2 SMTP 要点 ★

  • 使用 TCP，端口 25，可靠传输；
  • 直接传输：发送方邮件服务器（充当客户端）→ 接收方邮件服务器；
  • 传输三阶段：握手（问候）→ 消息传输 → 关闭；
  • 命令/响应交互（像 HTTP）：命令 ASCII 文本，响应有状态码和短语；
  • SMTP 使用持久连接；消息（首部+正文）必须是 7 位 ASCII；用 CRLF.CRLF 判断消息结束。
S: 220 hamburger.edu
C: HELO crepes.fr
S: 250 Hello crepes.fr, pleased to meet you
C: MAIL FROM: <alice@crepes.fr>
S: 250 alice@crepes.fr... Sender ok
C: RCPT TO: <bob@hamburger.edu>
S: 250 bob@hamburger.edu ... Recipient ok
C: DATA
S: 354 Enter mail, end with "." on a line by itself
C: Do you like ketchup?
C: .
S: 250 Message accepted for delivery
C: QUIT
S: 221 hamburger.edu closing connection

8.3 Alice 给 Bob 发邮件的六步 ★

  1. Alice 用用户代理撰写邮件发给 bob@someschool.edu；
  2. 用户代理用 SMTP 把消息发到 Alice 的邮件服务器，进入消息队列；
  3. Alice 服务器的 SMTP 客户端与 Bob 的邮件服务器建立 TCP 连接；
  4. 通过 TCP 连接发送 Alice 的消息；
  5. Bob 的邮件服务器把消息放入 Bob 的邮箱；
  6. Bob 调用用户代理读取消息。

8.4 SMTP vs HTTP ★必考对比

HTTPSMTP
模式客户端拉取（pull）客户端推送（push）
共同点都是 ASCII 命令/响应交互、有状态码
对象封装每个对象封装在自己的响应消息中多个对象放在一条多部分消息中

8.5 邮件消息格式与访问协议

  • RFC 2822 定义邮件消息语法：首部行（To:、From:、Subject:——注意与 SMTP 的 MAIL FROM:/RCPT TO: 命令不是一回事）+ 空行 + 正文（ASCII）；
  • 邮件访问协议（从服务器取邮件）：
    • SMTP 负责投递/存储到接收方服务器；
    • IMAP（Internet 邮件访问协议）：消息存在服务器上，提供检索、删除、文件夹管理；
    • HTTP：Gmail 等提供 Web 界面，底层发送用 SMTP、检索用 IMAP（或 POP）。

九、域名系统（DNS）

9.1 DNS 是什么 ★必背

  • 问题：主机有 IP 地址（32 位，用于寻址）和名称（如 cs.umass.edu，方便人类），两者如何映射？
  • DNS（Domain Name System，域名系统）：
    • 分布式数据库：由许多层级结构的名称服务器实现；
    • 应用层协议：主机与 DNS 服务器通信以解析名称（名称 ↔ 地址转换）；
    • 注意：DNS 是核心互联网功能，却作为应用层协议实现（复杂性放在网络边缘）。

9.2 DNS 服务 ★

  1. 主机名 → IP 地址转换（最主要）；
  2. 主机别名（规范名与别名）；
  3. 邮件服务器别名；
  4. 负载分配：一个名称对应多个 IP 地址（复制的 Web 服务器）。

9.3 为什么不集中式？★常考简答

无法扩展！具体原因：① 单点故障；② 流量太大（Comcast 一天 600 亿查询、Akamai 一天 2.2 万亿）；③ 远程集中式数据库距离远；④ 维护困难。

9.4 DNS 层次结构 ★必背

根 DNS 服务器（root）
    ↓
顶级域服务器（TLD：.com、.org、.edu、.cn、.uk …）
    ↓
权威 DNS 服务器（amazon.com、umass.edu 等组织自己的）
  • 查 www.amazon.com：先问根（得到 .com 服务器）→ 问 .com TLD（得到 amazon.com 服务器）→ 问 amazon 权威服务器（得到 IP）；
  • 根服务器：名称解析的"最后联络方"；全球 13 个逻辑根服务器，每个被多次复制；由 ICANN 管理；DNSSEC 提供安全；
  • TLD 服务器：负责 .com/.org/.edu 等及国家域名 .cn/.uk/.fr；
  • 权威服务器：组织自己的 DNS 服务器，提供本组织主机名→IP 的权威映射；
  • 本地 DNS 服务器★：不严格属于层次结构；主机的 DNS 查询先发给它；它返回缓存答案（可能过期！）或代为向层次结构查询；每个 ISP 都有（Windows 查看：ipconfig /all）。

9.5 迭代查询 vs 递归查询 ★必考辨析

迭代查询（iterated）递归查询（recursive）
方式被问到的服务器回复"该去问谁"（"我不认识，但你可以问这台"）把解析负担交给被查询的服务器，层层代查
负载本地 DNS 服务器自己挨个问高层服务器（根/TLD）负载过重

典型 8 步迭代流程：主机 → 本地 DNS → 根 → 本地 → TLD → 本地 → 权威 → 本地 → 主机。

9.6 DNS 缓存 ★

  • 任何名称服务器学到映射后都会缓存，查询时直接返回缓存映射；
  • 缓存条目在 TTL（Time To Live）后超时消失；
  • TLD 服务器通常缓存在本地 DNS 服务器中（因此根服务器不常被访问）；
  • 缓存可能过期：主机改了 IP，要等所有 TTL 到期全网才知道 → DNS 是尽力而为（best-effort）的名称解析。

9.7 DNS 资源记录（RR）★必背

RR 格式：(name, value, type, ttl)
typenamevalue

9.8 DNS 协议消息

  • 查询和响应消息格式相同；
  • 消息头：标识符（16 位，回复用相同编号）+ 标志（查询/回复、期望递归、递归可用、回复是否权威）；
  • 主体：问题区、回答 RRs、权威 RRs、附加信息。

9.9 注册域名（流程题）

新公司 networkutopia.com：① 在 DNS 注册商（如 Network Solutions）注册域名，提供权威服务器的名称和 IP；② 注册机构向 .com TLD 服务器插入两条记录：

(networkutopia.com, dns1.networkutopia.com, NS)
(dns1.networkutopia.com, 212.212.212.1, A)

③ 在本地建权威服务器，存 www 的 A 记录、邮件的 MX 记录。

9.10 DNS 安全（了解）

  • DDoS：轰炸根服务器（迄今未成功——流量过滤 + 本地缓存绕过根）；轰炸 TLD 服务器更危险；
  • 欺骗攻击：拦截查询返回假响应、DNS 缓存投毒；防御：DNSSEC（RFC 4033）。

10.3 易混概念辨析

概念对区别一句话
C/S vs P2PC/S 靠始终在线的服务器，扩展靠加机器；P2P 节点互为服务器，自我可扩展但难管理
TCP vs UDPTCP 可靠、面向连接、有流控拥控；UDP 不可靠、无连接、什么都不保证但轻量
非持久 vs 持久 HTTP非持久每对象一条连接（2RTT）；持久一条连接传多个对象（约 1RTT/对象）
GET vs POSTGET 数据放 URL ? 后；POST 数据放实体主体
HTTP vs SMTPHTTP 拉取（客户端要）；SMTP 推送（客户端送）；都是 ASCII 命令/响应
第一方 vs 第三方 cookie第一方来自你访问的网站；第三方来自没访问过的网站（跨站跟踪）
迭代 vs 递归查询迭代：被问者指路，自己继续问；递归：被问者代劳问到底
A vs CNAME vs NS vs MXA：名→IP；CNAME：别名→真名；NS：域→权威服务器；MX：域→邮件服务器
Web 缓存 vs 条件 GETWeb 缓存是机构里的代理服务器；条件 GET 是浏览器本地副本新鲜度检查（304）

10.4 一句话带过（非本次考点，了解即可）

  • 视频流与 CDN：视频占家庭 ISP 流量 80%；DASH = 服务器把视频切块多速率编码 + 客户端自适应选速率；CDN 把内容副本放到靠近用户的多个站点（Akamai 24 万台服务器）；Netflix 用 OpenConnect CDN + 清单文件。
  • 套接字编程实践：Python 中 UDP 用 socket(AF_INET, SOCK_DGRAM)，TCP 用 socket(AF_INET, SOCK_STREAM)，TCP 服务器需 bind/listen/accept。
